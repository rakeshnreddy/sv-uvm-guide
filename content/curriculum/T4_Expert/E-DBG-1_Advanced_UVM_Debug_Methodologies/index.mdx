## Quick Take
- **What it is:** A debugging playbook—structured logging, centralized event routing, selective waveform capture, and trace metadata—to tame large UVM environments.
- **Why it matters:** Scale brings opaque failures. Long SoC simulations or massive block-level regressions require rich, automated debug hooks to keep the signal-to-noise ratio high under tight schedules.
- **The Analogy:** Think of an airliner flight recorder: every test run should leave breadcrumbs that explain what the bench did and why, *before* you ever open a waveform viewer.

> **Signal:** When triage meetings begin with “rerun with more logs,” your methodology needs stronger built-in instrumentation.

## Build Your Mental Model

### 1. Structured Logging
Standard `uvm_info` is often treated as a dumping ground. Professional environments treat logs as machine-readable telemetry.
- **Standardize Categories:** Use consistent IDs (`ENV`, `DRV`, `SEQ`, `DBG`, `CFG`).
- **Correlation IDs:** Stamp sequence IDs, transaction handles, and timestamps into every message so logs join up clearly with coverage holes and scoreboard discrepancies.
- **Runtime Filtering:** Use `uvm_report_object::set_report_id_action_hier()` to promote or demote categories from the command line without recompiling.

### 2. The Centralized Debug Event Bus
Instead of hardcoding trigger logic deep inside individual monitors or scoreboards, advanced environments broadcast significant events to a centralized bus.

**Why add an event bus?**
- It centralizes debug events, allowing multiple subscribers (log formatters, telemetry exporters, trigger logic) to react without rewiring drivers or monitors.
- A scoreboard can simply broadcast `("SCB", "DATA_MISMATCH")`. A separate, specialized debug component listens for that tag and decides whether to dump waveforms, increment an error counter, or halt the simulation.

### 3. Selective Waveform Capture
Indiscriminately dumping waveforms for a 20-hour SoC regression is impossible. You must gate waveform dumping behind triggers.
- **Trigger-Driven Dumps:** Use the event bus to listen for errors or watchdog timeouts, and *then* fire vendor-specific waveform APIs (e.g., `$fsdbDumpvarsByFile`).
- **Ring Buffers:** Keep the last $N$ microseconds of history in memory and flush to disk only when a trigger fires, capturing the lead-up to the failure without the terabytes of idle traffic.

### 4. Trace Metadata Pipeline
- Emit JSON or NDJSON trace files mapping the test name to key lifecycle events (e.g., reset done, handshake stalled, scoreboard mismatch).
- Post-process these traces into dashboards so regression reports show high-level failure breadcrumbs before anyone opens a log file.

## Push Further: Triage Playbook

When an overnight regression turns red, follow this playbook rather than randomly adding displays:

1. **Check the Watchdog / Heartbeat:** Did a physical timeout occur, or did a `uvm_objection` hang? (See the Hang Lab for objection tracing).
2. **Review the Event Bus Trace:** Look for the final major events before the failure. Did the configuration phase complete? Did the scoreboard receive any items?
3. **Triggered Waveforms Only:** Open the specific selective-dump file created by the event bus trigger, rather than the full-run database.
4. **Reproduce Fast:** Divide and conquer. Reduce randomization seeds, trim the test duration, or bypass unrelated subsystems until the bug reproduces in under 60 seconds.
