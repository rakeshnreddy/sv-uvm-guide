## Quick Take

Portable Stimulus Standard (PSS) is an Accellera standard (PSS 2.0, IEEE P2851) that describes verification **scenarios as intent** rather than implementation. Tools compile PSS models into executable test content for simulation, emulation, FPGA prototyping, and post-silicon validation — all from the same specification.

> **Signal:** If your team maintains separate test suites for simulation and emulation, PSS can collapse them into one reusable model.

## How PSS Relates to UVM and Formal

| Concept | UVM Equivalent | Formal Equivalent | PSS Role |
|---|---|---|---|
| Stimulus | Sequences + constraints | Free inputs + assumptions | Intent actions + constraints |
| Ordering | [Virtual sequences](/curriculum/T2_Intermediate/I-UVM-3B_Advanced_Sequencing_and_Layering/virtual-sequences) | Not applicable | Activity graphs (DAGs) |
| Coverage | Covergroups + bins | `cover` properties | Coverage goals in the model |
| Checking | Scoreboards | `assert` properties | Assertions referenced or embedded |
| Portability | Simulation only | Formal-tool specific | All platforms from one model |

### Key differences from UVM sequences

1. **Declarative, not procedural:** PSS describes *what* should happen, not *how*. Tools resolve ordering and resource contention.
2. **Multi-platform:** The same PSS scenario can generate a constrained-random UVM sequence, an emulation boot script, or a bare-metal C test.
3. **Resource modeling:** PSS explicitly models shared resources (buses, memories, locks), enabling tools to schedule concurrent activities without deadlock.

## PSS Workflow Sketch

```
PSS Model (.pss)
    ├── Compile for simulation → UVM sequence + test
    ├── Compile for emulation  → Transaction-level API calls
    └── Compile for silicon    → Bare-metal C driver + checker
```

Each target platform gets a native test from the same specification. Coverage goals are also portable — the PSS model tracks which scenarios have been explored regardless of platform.

## When PSS Adds Value

- **Block-to-SoC reuse:** SoC-level integration tests that must run on multiple platforms.
- **Firmware-HW co-verification:** Scenarios involving both hardware transactions and firmware flows.
- **Multi-master arbitration:** Complex resource-sharing scenarios where hand-written sequences become fragile.
- **Regression portability:** One regression plan that tools can retarget as the DUT moves from simulation to emulation to silicon.

## Practical Adoption Considerations

1. **Tool dependency:** PSS compilation requires vendor-specific tools (Synopsys TestComposer, Cadence Perspec, Breker TrekSoC, etc.). Evaluate tooling before committing.
2. **Learning curve:** Teams accustomed to procedural UVM sequences need to shift to intent-based modeling. Plan for training investment.
3. **Integration with existing flows:** PSS does not replace UVM — it generates UVM code. Your existing agents, monitors, and scoreboards remain.
4. **Maturity:** PSS is still evolving. Start with well-bounded use cases (e.g., SoC boot sequences) before attempting full regression migration.

## Connection to Formal Verification

PSS constraints and formal assumptions share a conceptual foundation: both describe **legal behavior without specifying implementation**. Teams using both PSS and formal can:
- Extract formal assumptions from PSS resource constraints.
- Use formal analysis to prove that PSS-generated tests cover critical corner cases.
- Cross-reference PSS coverage goals with formal `cover` property results.
