## Quick Take
- **What it is:** Delta cycles are zero-time steps used to resolve event ordering locally. Race conditions occur when the order of execution between parallel blocks is non-deterministic.
- **Why it matters:** Improper use of blocking vs. non-blocking assignments is the #1 cause of simulation mismatches compared to synthesized hardware.
- **The Golden Rule:** Always use non-blocking (`<=`) for sequential logic (flip-flops) and blocking (`=`) for combinational logic.

## Build Your Mental Model

### Delta Cycles Explained
When multiple events trigger simultaneously (at the exact same simulation time step), the simulator uses delta cycles to process them sequentially. If signal `A` updates, triggering an `always` block that updates `B`, these happen in the same time step but separate delta cycles.

### The Race Condition Trap
A race condition happens when two `always` blocks read and write to the same variable at the same time using blocking assignments (`=`).

```systemverilog
module race_condition_demo;
  logic clk = 0;
  logic q1, q2, d;

  always #5 clk = ~clk;

  // BAD: Race Condition!
  always @(posedge clk) q1 = d;
  always @(posedge clk) q2 = q1;
  
  // Depending on which block evaluates first in the Active region,
  // q2 might get the OLD value of q1, or the NEW value of q1.
  // This is non-deterministic!
endmodule
```

### Blocking vs. Non-Blocking Assignments
To solve this, SystemVerilog provides non-blocking assignments (`<=`) for sequential logic (IEEE 1800-2023 §10.4).

```systemverilog
module safe_assignments_demo;
  logic clk = 0;
  logic q1, q2, d;

  always #5 clk = ~clk;

  // GOOD: Deterministic behavior
  always @(posedge clk) begin
    q1 <= d;  // Evaluates RHS now, updates LHS in NBA region
  end
  
  always @(posedge clk) begin
    q2 <= q1; // Evaluates RHS now (gets OLD q1), updates LHS in NBA region
  end
endmodule
```

<Alert variant="destructive">
**Common Bug:** Using blocking assignments (`=`) in sequential (clocked) `always_ff` blocks creates simulation race conditions that will NOT match the synthesized hardware. Always use `<=`.
</Alert>

## Make It Work
- **Strict Separation:** Never mix `=` and `<=` assignments to the same variable. It is illegal in many synthesis tools and causes unpredictable simulation behavior.
- **Testbench Synchronization:** When driving synchronous DUT signals from a testbench `initial` block, use non-blocking assignments (`<=`) synchronized to the clock edge, or use clocking blocks to completely abstract the race conditions away.

## Push Further
- Analyze what happens when you write to a variable from two different `always` blocks at the same time (even with non-blocking assignments). This is a multiple-driver conflict.
