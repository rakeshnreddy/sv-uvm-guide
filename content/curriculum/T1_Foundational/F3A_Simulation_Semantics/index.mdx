## Quick Take
- **What it is:** SystemVerilog simulation is event-driven. Time only advances when events occur.
- **Why it matters:** Understanding simulation semantics is crucial for avoiding race conditions and predicting hardware behavior accurately. If you don't understand how time flows, your testbenches will be fragile and simulator-dependent.
- **The Analogy:** Think of simulation time as a clock on the wall, while delta time is a stopwatch used to order immediate events before the wall clock ticks forward.

## Build Your Mental Model

### 1. Event-Driven Simulation
The simulator only evaluates code when an event triggers it. An event is typically a change in a signal's value. Between events, the simulator does absolutely nothing.

```systemverilog
module event_driven_demo;
  logic a, b;

  // This block only runs when 'a' changes.
  always @(a) begin
    b = ~a;
    $display("a changed to %b at time %0t", a, $time);
  end

  initial begin
    a = 0;
    #10 a = 1;
    #10 a = 0;
  end
endmodule
```

### 2. Time Steps and Timescale
Simulation time advances in discrete increments. Delay controls (`#`) explicitly advance time, but the meaning of `#10` depends on the active `` `timescale ``.

```systemverilog
`timescale 1ns/1ps // time_unit / time_precision

module time_steps_demo;
  initial begin
    $display("Time is %0t", $time); // 0
    #5.5; // Advances time by 5.5ns
    $display("Time is %0t", $time); // 5500 (ps resolution)
  end
endmodule
```

### 3. Simulation Time vs. Delta Time
Multiple events can occur at the exact same simulation time. The simulator processes these in "delta cycles" — zero-delay iterations that determine the order of evaluation without advancing the global simulation clock.

```systemverilog
module delta_time_demo;
  logic clk = 0;
  logic d, q1, q2;

  always #5 clk = ~clk;

  initial begin
    d = 1;
    @(posedge clk);
    // At posedge clk, q1 gets d, and q2 gets q1. 
    // This happens in delta cycles within the same simulation time step.
    q1 <= d;  // Delta cycle 1 (NBA region update)
    q2 <= q1; // Delta cycle 2 (subsequent evaluation triggered by q1 change)
  end
endmodule
```

## Make It Work
- Always define a `` `timescale `` at the top of your testbench files, or better yet, use `$timeformat` and pass timescales via compiler directives to avoid compilation order bugs.
- Recognize that `#0` delays are a legacy hack for forcing delta cycle delays. Do not use them in modern SystemVerilog; use non-blocking assignments (`<=`) or clocking blocks instead.

## Push Further
- Explore how the simulator engine actually processes events: it maintains an event queue sorted by time. When all events for the current time are exhausted, it advances to the next time in the queue.
- See how `$strobe` and `$monitor` leverage the Postponed region (end of the current time step) to print values only after all delta cycles have settled.
