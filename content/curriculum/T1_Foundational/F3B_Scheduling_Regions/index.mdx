## Quick Take
- **What it is:** The IEEE standard divides each simulation time step into multiple scheduling regions (Active, Inactive, NBA, Observed, Reactive, etc.).
- **Why it matters:** It prevents race conditions between testbench code and RTL code, and between blocking and non-blocking assignments.

## The SystemVerilog Scheduling Regions

SystemVerilog defines a strict order of execution within a single time step to prevent race conditions. The time step is divided into several regions. Here are the most critical ones you must understand for interviews:

| Region | Primary Purpose | Examples |
|---|---|---|
| **Preponed** | Sample stable values from the previous time step before any new changes occur. | Used by clocking blocks (input sampling) and concurrent assertions. |
| **Active** | Evaluate and update variables using blocking assignments, evaluate RHS of non-blocking assignments. | `a = b;`, `$display();` |
| **Inactive** | Process `#0` delaying statements (legacy, avoid using). | `#0 a = b;` |
| **NBA (Non-Blocking Assignment)** | Update LHS of non-blocking assignments evaluated in the Active region. | `a <= b;` |
| **Observed** | Evaluate concurrent assertions (SVA) using values sampled in the Preponed region. | `assert property (...)` |
| **Reactive** | Execute testbench program blocks and testbench checks. | Program block execution, action blocks of assertions |
| **Postponed** | Execute read-only tasks after all other regions have settled for the current time step. | `$monitor()`, `$strobe()` |

> [!IMPORTANT]
> **The Active vs. Reactive Split:** The original intent of SV scheduling was to isolate RTL (Active/NBA) from Testbench (Reactive). Testbenches driven from program blocks execute in the Reactive region, interacting with RTL via clocking blocks (sampled in Preponed, driven in Reactive) to guarantee zero race conditions. In modern UVM (which uses modules instead of programs), this is achieved purely via clocking block semantics and phasing.
