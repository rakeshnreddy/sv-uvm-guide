## Quick Take
- **What it is:** A standardized mechanism for exposing safe extension points within a UVM component's main execution loop, allowing test-specific behavior injection without subclassing.
- **Why it matters:** Without callbacks, every test-specific tweak (error injection, delay insertion, scoring hooks) requires a factory override and a dedicated subclass. Callbacks let you append, prepend, or stack multiple independent extensions on a single component instance—and toggle them at runtime.
- **The Standard:** IEEE 1800.2-2020 Clause 10.7 (`uvm_callback`) defines the callback base class and the `uvm_callbacks` parameterized type-wrapper for registration. Clause 10.8 (`uvm_callback_iter`) defines the iteration mechanism that components use internally to walk the callback queue.

## Build Your Mental Model

### The Three-Part Contract

Every callback scenario has exactly three participants:

1. **The callback interface** — a virtual class extending `uvm_callback` that declares hook methods (tasks or functions). The component developer writes this.
2. **The calling component** — registers the callback type with `` `uvm_register_cb `` and invokes hooks via `` `uvm_do_callbacks `` at chosen points in its execution loop.
3. **The test-specific implementation** — a concrete class extending the callback interface, overriding only the hooks it needs. The test writer creates and attaches this.

```systemverilog
// ──── Part 1: Callback interface (component developer) ────
virtual class my_driver_cb extends uvm_callback;
  // Called right before the driver wiggles pins
  virtual task pre_send(my_driver driver, ref packet pkt);
  endtask

  // Called right after driving completes
  virtual function void post_send(my_driver driver, const ref packet pkt);
  endfunction
endclass

// ──── Part 2: Calling component ────
class my_driver extends uvm_driver#(packet);
  `uvm_component_utils(my_driver)
  `uvm_register_cb(my_driver, my_driver_cb)

  function new(string name, uvm_component parent);
    super.new(name, parent);
  endfunction

  task run_phase(uvm_phase phase);
    packet pkt;
    forever begin
      seq_item_port.get_next_item(pkt);

      // Invoke all registered pre_send callbacks in order
      `uvm_do_callbacks(my_driver, my_driver_cb, pre_send(this, pkt))

      send_to_dut(pkt);

      // Invoke all registered post_send callbacks in order
      `uvm_do_callbacks(my_driver, my_driver_cb, post_send(this, pkt))

      seq_item_port.item_done();
    end
  endtask
endclass

// ──── Part 3: Test-specific callback ────
class inject_error_cb extends my_driver_cb;
  `uvm_object_utils(inject_error_cb)

  function new(string name = "inject_error_cb");
    super.new(name);
  endfunction

  virtual task pre_send(my_driver driver, ref packet pkt);
    `uvm_info("CB", "Injecting CRC error into packet", UVM_LOW)
    pkt.crc = ~pkt.crc;
  endtask
endclass
```

### Passing by Reference (`ref`) vs Handles

The `ref` keyword in a callback hook task changes the power dynamic significantly:
- **By value (default):** The callback can modify the *contents* of the transaction object through its handle. This is usually enough for field corruption, delay injection, or passive observation.
- **By `ref`:** The callback can *replace the entire transaction handle* in the driver's variable—substituting one object for another.
- **By `const ref`:** The callback can observe but not modify anything. Use this for post-action monitoring hooks.

Choose deliberately: `ref` should be reserved for hooks that genuinely need object-swap power.

### Callback Lifecycle and the Iteration Queue

When `` `uvm_do_callbacks `` executes, the UVM library instantiates a `uvm_callback_iter` object internally. This iterator walks the callback queue for the given `(component_type, callback_type)` pair **in FIFO registration order**—the first callback added executes first.

Key lifecycle rules:
- **Registration** uses `` `uvm_register_cb(comp_type, cb_type) `` inside the component class body. This is a compile-time type binding.
- **Attachment** uses `uvm_callbacks#(comp_type, cb_type)::add(comp_inst, cb_inst)` at runtime (typically in `build_phase`).
- **Removal** uses `uvm_callbacks#(comp_type, cb_type)::delete(comp_inst, cb_inst)`.
- **Iteration** happens each time `` `uvm_do_callbacks `` fires. The iterator walks the queue snapshot—modifications to the queue mid-iteration do not affect the current pass.

## Managing Callbacks at Runtime

### Adding and Removing

```systemverilog
// Add a callback to a specific driver instance
uvm_callbacks#(my_driver, my_driver_cb)::add(env.agent.driver, err_cb);

// Add a callback to ALL instances of my_driver (global)
uvm_callbacks#(my_driver, my_driver_cb)::add(null, err_cb);

// Remove a specific callback from a specific instance
uvm_callbacks#(my_driver, my_driver_cb)::delete(env.agent.driver, err_cb);
```

> **Ordering rule:** When multiple callbacks are attached to the same component instance, they execute in *FIFO registration order*. If you attach an error injector first and a delay injector second, errors are always injected before delays are added within the same hook invocation.

### Displaying Registered Callbacks

During debug, call `display()` to see all registered callbacks for a given type pair:

```systemverilog
// Print every callback attached to every my_driver instance
uvm_callbacks#(my_driver, my_driver_cb)::display();
```

This prints the callback queue order, instance name, and component path—essential for diagnosing "my callback is not firing" bugs.

## Callbacks vs Factory Overrides

The most common design question is: *Should I use a callback or a factory override?* They solve superficially similar problems but have fundamentally different architectural roles.

| Dimension | Factory Override | Callback |
|---|---|---|
| **Mechanism** | Replaces the *entire class* for a component or object | Appends/prepends behavior *around* existing logic |
| **Granularity** | One active override per type at any time | Multiple independent callbacks can be stacked |
| **Lifetime** | Set once during construction, persistent for the test | Can be added and removed dynamically at any phase |
| **Scope** | Structural—changes *what* the component *is* | Behavioral—changes *what the component does* at specific hooks |
| **Typical use** | Replacing a driver with a protocol variant, swapping the entire scoreboard | Injecting errors, adding delays, passive monitoring, per-test scoring hooks |
| **Overhead** | None at runtime (resolved at `create()` time) | Small per-invocation cost from `uvm_callback_iter` walking the queue |

**Rule of thumb:** If you need to *replace* the thing, use the factory. If you need to *augment* the thing (especially with multiple independent concerns), use callbacks.

## Debug & Failure Modes

### Common Pitfalls

1. **"My callback is not firing"**
   - Verify registration: did the component class declare `` `uvm_register_cb ``?
   - Verify attachment: did the test call `uvm_callbacks#(...)::add()` *before* the phase where the callback fires?
   - Verify type parameters: `uvm_callbacks#(wrong_driver_type, my_cb)::add(...)` will silently attach to a different type's queue.
   - Run `uvm_callbacks#(comp, cb)::display()` to inspect the queue.

2. **Callback ordering surprises**
   - Remember: callbacks execute in *add-order*, not alphabetical or priority order. If ordering matters, control the sequence of `add()` calls in your test's `build_phase`.

3. **Ghost callbacks after `delete()`**
   - If you `delete()` a callback mid-phase, it is removed from the queue for *future* `` `uvm_do_callbacks `` invocations. It will **not** be retroactively removed from an iteration already in progress.

4. **Performance in hot loops**
   - Each `` `uvm_do_callbacks `` invocation creates a `uvm_callback_iter` and walks the queue. In sub-nanosecond per-bit driver loops, this overhead can compound across millions of transactions. Profile before deploying callbacks in very tight loops.

### Debug Checklist

When a callback-related issue arises:
1. Call `uvm_callbacks#(T, CB)::display()` at the start of `run_phase`.
2. Add `uvm_info` traces inside callback hook methods to confirm they execute.
3. Verify the `ref`/value semantics—if your mutation is not visible, you may be editing a copy.
4. Check whether the callback was added to the correct *instance* vs `null` (all instances).

## Push Further: Advanced Patterns

### Stacking Independent Concerns

One of the most powerful patterns is stacking multiple independent callbacks on the same component:

```systemverilog
// In test build_phase:
inject_error_cb err_cb = inject_error_cb::type_id::create("err_cb");
add_delay_cb    dly_cb = add_delay_cb::type_id::create("dly_cb");
scoreboard_cb   scb_cb = scoreboard_cb::type_id::create("scb_cb");

uvm_callbacks#(my_driver, my_driver_cb)::add(env.agent.driver, err_cb);
uvm_callbacks#(my_driver, my_driver_cb)::add(env.agent.driver, dly_cb);
uvm_callbacks#(my_driver, my_driver_cb)::add(env.agent.driver, scb_cb);

// All three fire in order at each `uvm_do_callbacks` invocation.
// Each is independently removable without affecting the others.
```

This composability is the key architectural advantage over factory overrides—you cannot cleanly stack three independent concerns using a single subclass chain.

### Dynamic Toggling During `run_phase`

Callbacks can be added and removed at any time, enabling scenario-based injection:

```systemverilog
task run_phase(uvm_phase phase);
  phase.raise_objection(this);

  // Run clean traffic first
  normal_seq.start(env.agent.sqr);

  // Now inject errors for the next batch
  uvm_callbacks#(my_driver, my_driver_cb)::add(env.agent.driver, err_cb);
  error_seq.start(env.agent.sqr);

  // Remove and run clean again
  uvm_callbacks#(my_driver, my_driver_cb)::delete(env.agent.driver, err_cb);
  recovery_seq.start(env.agent.sqr);

  phase.drop_objection(this);
endtask
```
