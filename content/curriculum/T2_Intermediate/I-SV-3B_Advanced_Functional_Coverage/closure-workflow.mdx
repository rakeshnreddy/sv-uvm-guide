## Level 1: The Infinite Loop Warning
If you write a completely unconstrained random test and let it run forever, will you hit 100% coverage? 

Mathematically, perhaps. But in reality, your simulation will run out of time, or the odds of hitting a specific 3-event corner case naturally are 1 in a billion. You cannot achieve closure purely by running more random seeds. You must actively participate in a **Closure Loop**.

## Level 2: The Canonical Workflow

Coverage closure is a repetitive engineering process:

1. **Run Regressions:** Run your test suite with many random seeds.
2. **Merge Databases:** The CI/CD system merges all the individual test coverage databases (UCDB/UCIS) into one master database.
3. **Identify Holes:** Open the master database in a viewer. Look for bins with 0 hits.
4. **Analyze the Unhit Bins:** Why wasn't this bin hit?
   - Is it impossible? (Write a `ignore_bins` or fix the V-Plan).
   - Is it legal, but the generator constraints make it extremely rare?
   - Is the testbench missing a directed sequence to set up the precise conditions?
5. **Adjust Stimulus:** Change constraint weights (`dist`), write a new directed sequence, or create a specific test to target the hole.
6. **Repeat:** Re-run the regression and verify the hole is filled.

## Level 3: Practice Makes Perfect

Reading about closure is easy; actually doing it requires an intuition for interpreting coverage data and translating it back into generator constraints. 

In the upcoming lab, you will be handed a mostly-working ALU testbench. You will run it, analyze the coverage holes in the report, and then modify the constraints using `dist` weighting and edge-case generation to hit 100% without rewriting the whole environment.
