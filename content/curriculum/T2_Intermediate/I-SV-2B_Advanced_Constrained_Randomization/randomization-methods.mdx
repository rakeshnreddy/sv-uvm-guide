## Level 1: Beyond Classes
*IEEE 1800-2023, Clause 18.12: std::randomize() and Clause 18.16: randcase/randsequence*

While `obj.randomize()` is the most common way to generate random data, sometimes you need to randomize local variables within a function or randomly choose an execution path without defining a whole new class. SystemVerilog provides procedural randomization tools for exactly these scenarios.

## Level 2: Core Mechanics

### 1. `std::randomize()`
The `std::randomize()` function allows you to randomize any variable in the current scope. It works just like `obj.randomize()`, and you can even attach inline constraints using `with`.

<InteractiveCode language="systemverilog" code={`module test;
  initial begin
    int unsigned payload_size;
    bit [7:0]  start_addr;

    // Randomize local variables with inline constraints
    if (!std::randomize(payload_size, start_addr) with {
      payload_size inside {[64:1500]};
      start_addr < 8'h20;
    }) begin
      $fatal(1, "Randomization failed!");
    end
    
    $display("Generated size: %0d, addr: %0h", payload_size, start_addr);
  end
endmodule`} />

### 2. `randcase`
`randcase` is a `case` statement that makes weighted random choices between execution branches. The expression before the colon is the weight.

<InteractiveCode language="systemverilog" code={`task drive_traffic();
  repeat(10) begin
    randcase
      3: send_read();    // 3/10 or 30% probability
      5: send_write();   // 5/10 or 50% probability
      2: send_reset();   // 2/10 or 20% probability
    endcase
  end
endtask`} />

### 3. `randsequence`
`randsequence` is a powerful construct for generating random token sequences, often used for instructional streams or assembly generation. It looks like a formal grammar definition.

```systemverilog
randsequence( main )
  main  : setup execute cleanup;
  setup : reset | init;          // Equal probability choice
  execute : read := 3 | write := 7; // Weighted choice
  reset : { $display("RESET"); };
  init  : { $display("INIT"); };
  read  : { $display("READ"); };
  write : { $display("WRITE"); };
  cleanup : { $display("CLEAN"); };
endsequence
```

## Level 3: Reproducibility and Seed Stability

A testbench that is "random" but cannot be reproduced is a liability. You must be able to reproduce failures.

### The Seed
Every random variable stream in SV is deterministic if you start with the same initial seed. Simulators accept command-line arguments (like `+ntb_random_seed=1234` in VCS or `-sv_seed 1234` in Xcelium) to set the master seed.

### Debugging Flaky Bugs
1. **Always log the seed:** Ensure your testbench prints the master random seed at the very beginning of the test.
2. **Replay the exact seed:** To debug a random failure, re-run the exact same test with the exact same seed from the log.
3. **Beware of thread stability:** If you spawn random threads (`fork...join_none`) in a non-deterministic order (e.g., based on real-time events that change slightly), the random number generator will be drawn from in a different order, and the same global seed will produce a different scenario. Keep random generation in deterministic, strictly ordered threads!
