  ## Level 1: The Big Idea
  *IEEE 1800-2023, Clause 15.4: Mailboxes*

  A **mailbox** is a built-in message queue. One process `put`s an item into the mailbox while another `get`s it out. It's like dropping a letter into a box for someone else to pick up later.

  ## Level 2: Practical Usage

  ### Basic Operations

  | Call | Blocking? | Use it when |
  | --- | --- | --- |
  | `put(item)` | Blocks if the mailbox is full | Backpressure producer threads until consumers catch up. |
  | `try_put(item)` | Returns 0 if full | Drop low-priority work instead of stalling. |
  | `get(item)` | Blocks until something arrives | Classic producer/consumer handshakes. |
  | `try_get(item)` | Returns 0 if empty | Poll without blocking a monitor. |
  | `peek(item)` | Non-destructive read | Look at the next transaction while leaving it queued. |
  | `num()` | Instant | Inspect queue depth for assertions or adaptive throttling. |

  <InteractiveCode>
  ```systemverilog
  typedef struct { int id; bit [7:0] data; } packet_t;
  mailbox #(packet_t) mbx = new(4); // depth-limited queue

  task automatic produce_packet(int id);
    packet_t p = '{id: id, data: $urandom_range(0, 255)};
    if (!mbx.try_put(p)) begin
      // Backpressure: wait for space, then block on put
      wait (mbx.num() < 4);
      mbx.put(p);
    end
  endtask

  initial begin
    for (int idx = 0; idx < 8; idx++) begin
      produce_packet(idx);
    end
  end

  initial begin
    packet_t pkt;
    forever begin
      mbx.get(pkt);
      $display("[%0t] Consumer saw id=%0d depth=%0d", $time, pkt.id, mbx.num());
    end
  end
  ```
  </InteractiveCode>

  Mailboxes often carry randomized transactions. For generating those transactions, see [Constrained Randomization](/curriculum/T2_Intermediate/I-SV-2A_Constrained_Randomization_Fundamentals).

  ## Failure Modes: Leaks and Backpressure

  **The Infinite Queue (Memory Leak):** 
  Calling `new()` with no arguments creates an unbounded (infinite) mailbox. If your producer thread (e.g., a fast transaction generator) pushes items faster than your consumer thread (e.g., a slow APB driver) can pull them, the mailbox will grow infinitely. This will eventually exhaust the simulator's memory and crash the process. 
  
  **The Fix (Backpressure):** Always use bounded mailboxes `new(DEPTH)` for driver queues. When the mailbox is full, the producer's `put()` will naturally block, creating *backpressure* that forces the generator to wait for the hardware to catch up.

  **Mailboxes vs. TLM/FIFOs:** 
  In modern UVM environments, raw SystemVerilog mailboxes are rarely used. UVM’s `uvm_tlm_fifo` wraps a mailbox with analysis ports, standard reset control, and comprehensive blocking/non-blocking APIs. You should transition to TLM FIFOs when bridging IPC into UVM testbench architecture.

  **One-to-one only:** Mailboxes always deliver each message to a single consumer. When you need broadcast semantics, rely on [analysis ports](/curriculum/T2_Intermediate/I-UVM-2B_TLM_Connections#monitors-scoreboards-and-the-analysis-fabric) or register callbacks.
